Formula Guide

Formulas let your templates do the math automatically — so totals, deadlines, and calculations fill in on their own without anyone typing them manually.

What is a formula?

A formula is an instruction you write once in a field, telling the system how to calculate its value based on other fields. Once set up, it runs automatically every time someone fills in the template.

💡
Think of it like a calculator. Instead of someone manually typing "Total = Item 1 + Item 2 + Item 3", you write the formula once and the system calculates it for every submission.

Two types of formula fields

TypeWhat it calculatesExample use
Number Amounts, totals, counts, percentages Total cost, tax amount, item count
Date Deadlines, durations, time gaps Due date, review deadline, days remaining

How to write a formula

A formula always follows a predictable structure. Once you learn the pattern, all formulas work the same way.

ROUND ( {Unit Price} × {Quantity} , 2 )
Function name
Field reference (your template's field labels)
A plain number
Operator or separator
✅
You don't need to memorize all of this. In the formula editor, you can click on a field to insert it automatically. The curly braces {} are added for you.

Referencing your fields

To use a field's value in a formula, wrap its label in curly braces: {Field Label}.

Your field labelHow to reference it
Unit Price{Unit Price}
Release Date{Release Date}
Number of Items{Number of Items}
Approved Budget{Approved Budget}
✅
Always use the field picker. Click a field in the picker to insert its reference automatically — you never need to type curly braces yourself. This also prevents typos that would break the formula.
📋
Master list fields work like any other field. If your template has a dropdown linked to a master list, both the dropdown field and any autofill fields it populates can be referenced in formulas using the same {Field Label} syntax — no special treatment needed. Once the user makes a selection and autofill runs, those values are available to your formulas immediately. If the user changes their selection, autofill fields update and any dependent formulas recalculate automatically.

When two fields have the same name

Templates can have fields with the same label — for example, "Amount" appearing in multiple sections. When that happens, the formula needs a way to tell them apart. A number suffix is added automatically.

SituationWhat you see in the pickerWhat gets inserted
Only one field named "Amount" Amount {Amount}
Two fields named "Amount" Amount #1   Amount #2 {Amount#1}   {Amount#2}
Three fields named "Amount" Amount #1   Amount #2   Amount #3 {Amount#1}   {Amount#2}   {Amount#3}
Example — adding two "Amount" fields from different sections
{Amount#1} + {Amount#2}
Hover over {Amount#1} or {Amount#2} in the formula editor to see a tooltip showing which section each field belongs to.
Tooltip on Amount #1
{Amount#1}
Section: Travel Expenses
Tooltip on Amount #2
{Amount#2}
Section: Meal Expenses
💡
The numbers don't change when you reorder sections. #1 and #2 are assigned in the order the fields were created, not the order they appear on screen. Moving sections around will never break an existing formula.
⚠️
The suffix only appears when there's a conflict. If your template has only one field named "Amount", it stays as {Amount} — no suffix. The suffix appears the moment a second field with the same name is added.

Renaming a field that's used in a formula

If you rename any field that is referenced in a formula, the formula updates automatically to match the new name. You don't need to edit the formula yourself.

Example — renaming a field
Before rename — formula reads:
{Unit Price} * {Quantity}
You rename Unit Price → Price Per Item. After save, formula automatically becomes:
{Price Per Item} * {Quantity}

This applies to all rename cases — including when a suffix is added or dropped:

What changedFormula beforeFormula after
Rename unique field {Unit Price} {Price Per Item}
Add a second "Amount" field (conflict created) {Amount} {Amount#1}
Remove or rename the duplicate so conflict is resolved {Amount#1} {Amount}
⚠️
Renaming always updates the formula immediately. There is no undo for this. If you rename a field and then rename it back, the formula will reflect both changes.

MIN and MAX

MIN and MAX compare two values and return the smaller or larger one. Useful for setting caps or floors on calculated values.

MAX
Return the larger of two values
e.g. enforce a minimum charge
MAX({Calculated Fee}, 500) ← always at least 500
Calculated Fee = 200
→ 500 (minimum applied)
Calculated Fee = 1,200
→ 1,200 (above minimum)
MIN
Return the smaller of two values
e.g. cap a value so it never exceeds a limit
MIN({Requested Amount}, 10000) ← never more than 10,000
Requested = 8,000
→ 8,000
Requested = 15,000
→ 10,000 (cap applied)

IF — conditional formulas

IF lets you return different values depending on a condition. If the condition is true, it returns the first value; if false, it returns the second.

IF ( condition , value if true , value if false )
Read it as: "IF [this is true], THEN [do this], OTHERWISE [do this]."
Apply a discount if quantity is high
IF({Quantity} >= 10, {Unit Price} * 0.9, {Unit Price})

If Quantity is 10 or more → apply 10% discount. Otherwise → use full price.

Qty = 5, Price = 1,000
→ 1,000 (no discount)
Qty = 12, Price = 1,000
→ 900 (10% off)
Flag if over budget
IF({Total Expenses} > {Approved Budget}, "Over budget", "Within budget")
Expenses = 8,000, Budget = 10,000
→ "Within budget"
Expenses = 11,000, Budget = 10,000
→ "Over budget"

Adding and subtracting days

DATEADD shifts a date forward or backward by a given number of units. Use it to calculate deadlines, review dates, or expiry dates.

DATEADD ( {date field} , number of days , "days" )
Deadline = 14 days after release date
DATEADD({Release Date}, 14, "days")
Release Date = Jan 1
→ Jan 15
Release Date = Mar 20
→ Apr 3
Review date = 7 days before the deadline
DATEADD({Deadline}, -7, "days") ← use a negative number to go back in time
Deadline = Apr 30
→ Apr 23
Deadline = Dec 31
→ Dec 24
UnitWrite asExample
Days"days"DATEADD({Start}, 7, "days")
Weeks"weeks"DATEADD({Start}, 2, "weeks")
Months"months"DATEADD({Start}, 1, "months")

Days between two dates

DATEDIF calculates the difference between two dates. The result is a number — use it in a number formula field.

Project duration in days
DATEDIF({Start Date}, {End Date}, "days")
Start = Jan 1, End = Jan 15
→ 14 days
Start = Mar 1, End = Apr 1
→ 31 days

TODAY()

TODAY() returns today's date. It updates automatically — so formulas using TODAY() will recalculate every day.

📌
TODAY() is ideal for deadline calculations. Use it anywhere you need "from now" or "until now" logic — days remaining, days overdue, and so on.
Days between submission date and a deadline
DATEDIF(TODAY(), {Deadline}, "days")
Submitted Jan 1, Deadline Jan 10
→ 9 days remaining at submission
Submitted Jan 12, Deadline Jan 10
→ -2 (submitted 2 days late)
Auto-set review date to 30 days from submission
DATEADD(TODAY(), 30, "days")

The review date is set to 30 days from the submission date — no manual input needed. The result stays fixed after submission.

Multi-entry sections

A multi-entry section lets the person filling the form add multiple rows — like a list of items in an order, or a list of tasks. Formulas inside the section calculate per row. Formulas outside can aggregate across all rows.

💡
Think of it like a table in Excel. Each row is one set of filled-in fields. A row-level formula calculates something for that row. An aggregate formula totals or summarises across all rows.

Row-level formula

A formula inside a multi-entry section runs once for each row. It references sibling fields — other fields in the same row — directly by name.

Aggregate formula

An aggregate formula lives outside the multi-entry section and summarises values across all rows. It uses dot notation to reference fields inside the section.

⚠️
Dot notation is required when referencing a multi-entry section from outside it. Writing {Subtotal} alone is ambiguous if that label exists elsewhere in the template. Always use {Items.Subtotal} to be specific about which section you mean.

Combining row-level and aggregate formulas

The most common pattern is to use both layers together: a row-level formula calculates each row's value, then an aggregate formula totals them up.

Full example — expense form
Step 1 — Row-level formula inside "Items" multi-entry section
Field: Subtotal {Price} * {Quantity}
Step 2 — Aggregate formula in a separate "Summary" section
Field: Grand Total SUM({Items.Subtotal})
Step 3 — Optional: use Grand Total in another formula
Field: Tax (10%) {Grand Total} * 0.1
✅
Aggregate formulas update in real time. As the person filling the form adds or removes rows, any SUM, COUNT, AVERAGE, MIN, or MAX referencing that section recalculates automatically.

Common errors and how to fix them

Some errors happen while writing a formula. Others happen later, when someone edits the template. Both are covered here.

Errors while writing a formula

ErrorMost likely causeHow to fix
Field not found The field label inside { } doesn't match any field in your template — usually a typo or capitalization mismatch Use the field picker to insert references safely — it eliminates this error entirely.
Wrong field type You're using a date field in a number formula, or a number field in a date formula. Also triggered if you reference a text field (e.g. a dropdown or short answer) in any formula Check the field type in your template. Number formulas need number fields; date formulas need date fields. Text fields cannot be used in formulas.
Missing parenthesis An opening ( doesn't have a matching closing ) Count your parentheses — every ( needs a ). Nested functions like IF(ROUND(...)) need two closing parentheses at the end.
Divide by zero The field being divided by is blank or contains 0 at the time of submission Wrap the division in an IF to guard against it: IF({Count} = 0, 0, {Total} / {Count})
Circular reference Field A's formula references Field B, and Field B's formula references Field A — creating a loop with no end Open one of the two formula fields and remove the reference that creates the loop. A formula field cannot reference another formula field that references it back.
Result is blank One or more referenced fields haven't been filled in yet by the person submitting the form This is expected — formulas only calculate once all referenced fields have values. The result fills in automatically when they do.

Errors caused by editing the template later

These errors don't appear when you first write the formula — they're triggered later when someone modifies the template structure.

ErrorWhat caused itHow to fix
Field was deleted A field that was referenced in a formula has been deleted from the template. This is different from "field not found" — the field existed before and was intentionally removed. You'll see a warning before deletion if the field is used in a formula. If it's already deleted, open the formula and either remove the broken reference or replace it with a different field.
Field type changed A field referenced in a formula had its type changed — for example, from a number field to a text field, or from a date field to a dropdown. The formula no longer receives the type of value it expects. You'll see a warning before a type change if the field is used in a formula. If the change has already been made, open the formula and update it to use a compatible field, or change the field type back.
💡
Warnings always appear before a destructive change. Deleting a field or changing its type will show you a list of affected formulas before the action is confirmed — so nothing breaks without you knowing about it first.

Implementation Notes
Open questions and decisions that need to be resolved before development. Each item below has a recommended approach and a reason.
Decision Required
Field reference storage — ID vs. label string
How are field references stored internally in a saved formula? This single decision affects nearly every other item on this list.
Option A — Store as label string
{Unit Price} * {Quantity}
Simple to implement. Breaks on any field rename. Japanese labels require additional parser work. Duplicate field names require suffix logic baked into the parser.
Option B — Store as stable ID ✦ Recommended
{field_a3k} * {field_b7m}
Parser works off internal IDs. Display layer shows label. Rename, reorder, Japanese labels — all irrelevant to the parser. Duplicate handling and suffix logic are display-only concerns.
Recommendation: Option B. The formula editor displays {Unit Price} to the user but stores {field_a3k} internally. The field picker handles the ID-to-label mapping on insert. This eliminates rename breakage, reorder breakage, and most Japanese encoding issues in one decision.
Decision Required
Duplicate field name display — suffix assignment
When two fields share the same label, a #1 / #2 suffix is shown in the field picker and formula editor. The suffix must be stable across section reorders.
Rules:
1. Suffix is assigned by field creation order, not display order.
2. Suffix is scoped per label — Amount #2 is the 2nd field named "Amount", not the 2nd field overall.
3. Suffix appears only when a conflict exists. A unique field stays as {Amount}.
4. When a conflict is resolved (rename or delete), suffix drops automatically and affected formulas update.
Edge case: user renames a field to a label that creates a new conflict. System should detect this and warn before confirming — or auto-assign the next available suffix. Do not allow silent conflicts.
Decision Required
Rename and type-change guards
Any action that modifies a field referenced in a formula must show a warning listing affected formulas before the action is confirmed.
Actions that require a pre-confirmation warning:
— Field rename (including suffix changes from conflict resolution)
— Field type change (e.g. number → text breaks any formula using that field)
— Field deletion
— Section rename (if section dot notation is used for multi-entry section references)
The warning should name each affected formula field, not just say "some formulas will be updated." Users need to know exactly what changes.
Japanese Support
Multilingual template support
Templates may be built entirely in Japanese. The formula system must handle non-ASCII field labels without errors.
Known risks if using label-string storage (Option A):
— Full-width spaces (　U+3000) vs half-width spaces ( U+0020) in field labels cause silent mismatch
— Full-width curly braces ｛｝ vs half-width {} — different Unicode characters, parser won't match
— UTF-8 must be enforced end-to-end: input field → formula storage → evaluation engine
If using stable ID storage (Option B — recommended):
Japanese label text is irrelevant to the parser. The label is display-only. Full-width/half-width normalisation is still recommended for the field picker search/filter UX, but it cannot break formulas.
Do not ask users to use underscores instead of spaces in Japanese field names. This is a parser implementation concern, not a naming convention to impose on template builders.
Multi-entry Sections
Dot notation for cross-section references
Aggregate formulas outside a multi-entry section reference fields inside it using dot notation: {SectionName.FieldName}. The section identifier in this notation needs a stable reference for the same reason field references do.
Questions to resolve:
1. Is the section identifier in {Items.Subtotal} the section's display name or a stable ID? Recommend stable ID for the same reasons as field references.
2. Section names can be duplicated — same disambiguation problem as fields. Recommend the same suffix approach (#1/#2) applied to section names in dot notation.
3. If a section is renamed, any formula using {OldName.FieldName} must update automatically — same rename guard logic as field renames.
Scope note: row-level formulas (inside the multi-entry section) reference sibling fields directly without dot notation — no section identifier needed. Dot notation is only required for aggregate formulas referencing a multi-entry section from outside it.
Confirm with team
Master list — value storage at submission time
When a user selects a value from a master list dropdown, the autofill fields populate from the corresponding row. The question is when those values get locked.
Recommended behaviour:
— Autofill values are stored as a snapshot at submission time
— If the master list is updated after submission, existing submitted values do not change
— Formulas that reference autofill fields calculate against the stored snapshot, not the live master list
Confirm this is the intended behaviour before development. If autofill values are re-fetched live from the master list on every form load, formula results could change after submission — which would be unexpected to users reviewing past entries.
